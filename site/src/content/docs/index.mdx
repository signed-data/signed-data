## Problem

<div class="card-section">

Real-time data feeds have a provenance gap. When a JSON payload arrives, you trust the
transport, the intermediaries, and the source all at once. There is no cryptographic proof
the data is authentic, unmodified, or from who it claims to be from. At the same time,
identities in feeds are opaque strings — LLMs need data that is both machine-readable
*and* human-understandable.

</div>

## What we build

<div class="card-section">

**[CDS — Curated Data Standard](/docs/spec/v0-2-0/)** — a universal signed JSON-LD envelope
for real-time data. Every identity is a dereferenceable HTTP URI; every event carries an
embedded `context.summary` so data and meaning travel together.

**[SDKs](/docs/getting-started/)** in Python and TypeScript that produce, sign, and
verify CDS events with deterministic RSA-PSS SHA-256.

**[MCP servers](/products/)** that expose signed Brazilian finance, commodities, companies,
and lottery data as tools for Claude and other agents.

</div>

## Principles

<div class="card-section">

- **Standard is the contract.** CDS and SDKs are infrastructure-agnostic; anyone can run ingestors and operators.
- **Provenance, not only TLS.** HTTPS secures the channel; signatures anchor the payload.
- **5-star Linked Data.** URI-based identities, JSON-LD envelopes, dereferenceable everything.
- **LLM-ready.** Events include a `context.summary` so data and meaning travel together.
- **Curated.** Structured sources, registered and fingerprinted — no unverified scraping.

</div>
