## Problema

<div class="card-section">

Los feeds de datos en tiempo real tienen una brecha de procedencia. Cuando llega un payload JSON,
confías en el transporte, en los intermediarios y en la fuente todo a la vez. No hay prueba
criptográfica de que los datos sean auténticos, no modificados o de quien dicen provenir. Al mismo tiempo,
las identidades en los feeds son cadenas opacas — los LLMs necesitan datos que sean a la vez legibles
por máquina *y* comprensibles para humanos.

</div>

## Lo que construimos

<div class="card-section">

**[CDS — Curated Data Standard](/docs/spec/v0-2-0/)** (Estándar de Datos Curados) — un envoltorio JSON-LD firmado universal
para datos en tiempo real. Cada identidad es un URI HTTP dereferenciable; cada evento lleva un
`context.summary` embebido para que los datos y el significado viajen juntos.

**[SDKs](/docs/getting-started/)** en Python y TypeScript que producen, firman y
verifican eventos CDS con RSA-PSS SHA-256 determinista.

**[Servidores MCP](/products/)** que exponen datos firmados de finanzas, commodities, empresas
y loterías brasileñas como herramientas para Claude y otros agentes.

</div>

## Principios

<div class="card-section">

- **El estándar es el contrato.** CDS y los SDKs son agnósticos de la infraestructura; cualquiera puede operar ingestores y operadores.
- **Procedencia, no solo TLS.** HTTPS asegura el canal; las firmas anclan el payload.
- **Linked Data de 5 estrellas.** Identidades basadas en URI, envoltorios JSON-LD, todo dereferenciable.
- **Listo para LLMs.** Los eventos incluyen un `context.summary` para que los datos y el significado viajen juntos.
- **Curado.** Fuentes estructuradas, registradas y con fingerprint — sin scraping no verificado.

</div>
