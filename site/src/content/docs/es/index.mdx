## Problema

<div class="card-section">

Cuando recibes un JSON de un feed de terceros, confías en el transporte, los
intermediarios y la fuente al mismo tiempo. No hay prueba de que el payload sea
exactamente lo que fue publicado, por quién ni cuándo.

</div>

## Lo que construimos

<div class="card-section">

**[cds](https://github.com/signed-data/cds)** — el
**Curated Data Standard**: un sobre universal para datos en tiempo real
(clima, deportes, lotería, noticias, …), firmado con RSA-PSS, con un resumen incorporado.

**[mcp-lottery](https://github.com/signed-data/mcp-lottery)** —
Servidor MCP que expone sorteos de la lotería brasileña como herramientas para Claude y otros agentes.

</div>

## SDKs

<div class="card-section">

| Lenguaje     | Paquete               | Instalación                       |
| ------------ | --------------------- | --------------------------------- |
| Python 3.12+ | `signeddata-cds`      | `pip install signeddata-cds`      |

</div>

## Principios

<div class="card-section">

- **El estándar es el contrato.** CDS y SDKs son agnósticos de infraestructura; cualquiera puede operar ingestores y operadores.
- **Procedencia, no solo TLS.** HTTPS protege el canal; las firmas anclan el payload.
- **Listo para LLMs.** Los eventos incluyen un `context.summary` para que los datos y el significado viajen juntos.
- **Curado.** Fuentes estructuradas, registradas y con fingerprint — sin scraping no verificado.

</div>
