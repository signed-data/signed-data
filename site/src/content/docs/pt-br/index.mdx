## Problema

<div class="card-section">

Feeds de dados em tempo real têm uma lacuna de proveniência. Quando um payload JSON chega, você confia no
transporte, nos intermediários e na fonte de uma só vez. Não há prova criptográfica
de que o dado é autêntico, não modificado, ou de quem ele alega ser. Ao mesmo tempo,
identidades em feeds são strings opacas — LLMs precisam de dados que sejam tanto legíveis por máquina
*quanto* compreensíveis por humanos.

</div>

## O que construímos

<div class="card-section">

**[CDS — Curated Data Standard (Padrão de Dados Curados)](/docs/spec/v0-2-0/)** — um envelope universal JSON-LD assinado
para dados em tempo real. Toda identidade é uma URI HTTP dereferenciável; todo evento carrega um
`context.summary` embutido para que dados e significado viajem juntos.

**[SDKs](/docs/getting-started/)** em Python e TypeScript que produzem, assinam e
verificam eventos CDS com RSA-PSS SHA-256 determinístico.

**[Servidores MCP](/products/)** que expõem dados assinados brasileiros de finanças, commodities, empresas
e loterias como ferramentas para Claude e outros agentes.

</div>

## Princípios

<div class="card-section">

- **O padrão é o contrato.** CDS e SDKs são agnósticos de infraestrutura; qualquer um pode rodar ingestores e operadores.
- **Proveniência, não apenas TLS.** HTTPS protege o canal; assinaturas ancoram o payload.
- **Linked Data 5 estrelas.** Identidades baseadas em URI, envelopes JSON-LD, tudo dereferenciável.
- **Pronto para LLM.** Eventos incluem um `context.summary` para que dados e significado viajem juntos.
- **Curado.** Fontes estruturadas, registradas e com fingerprint — sem scraping não verificado.

</div>
